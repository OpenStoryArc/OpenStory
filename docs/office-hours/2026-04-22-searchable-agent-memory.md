## Problem Statement

Engineers building seriously with AI coding agents (Claude Code, Cursor, pi-mono, etc.) are losing trust in their own workflow because the agent's session data is ephemeral, opaque, and owned by the platform. When a session expires, when context resets, when you switch machines, when you need to defend a decision in code review — the data you need is either gone or buried in a place you can't search.

The deeper problem is **trust**. You can't pick up where you left off if you can't see what happened. You can't teach what you can't show. You can't advocate without evidence. And the entropy this creates leaks into both the system and the team dynamics around it (Conway's Law).

OpenStory captures every agent session as portable CloudEvents in a store the user owns, then makes that history searchable, observable, and shareable.

## Demand Evidence

Honest grade: **low but not zero, and the strongest signal is behavioral.**

- **Founders use it daily:** Katie and Max (both enterprise architects/engineers) run OpenStory across 4 personal machines, with active sessions live as of this writing.
- **Cross-state real-time co-observation has emerged spontaneously.** Max watches Katie's sessions from another state in real-time, "for fun and for trust." This is a behavior the product enabled that didn't exist before — the closest thing to organic pull this session surfaced.
- **Friends and family are using it** (small N, doesn't count as external demand).
- **No external paying users.** No verified non-friend users.
- **Procurement-blocked enterprise pull:** Both founders' enterprise financial-services client desperately needs this but cannot deploy it on a startup-relevant timeline (12+ months for new tooling). This is a *reason this can't be an enterprise sale* — not demand evidence.

The demand-validation plan (see Distribution Plan below) is the actual experiment. Until 5 people outside the building install and 2 are still using it a week later, demand is unverified.

## Status Quo

What an AI-native engineer does today, without OpenStory:

- **Scrolls Claude Code chat history** trying to reconstruct what happened in a 4-hour session for a PR description. Gives up, writes a vague summary.
- **Loses session state when context resets** (compaction, /clear, machine switch, server-side expiry). Spends 5-15 minutes manually re-priming the new session via copy-paste, CLAUDE.md edits, or `/remember`.
- **Cannot find a solution from 3 weeks ago** that they know they discovered with the agent. Re-derives it. Or asks the agent to figure it out again, paying tokens twice.
- **Cannot show a junior engineer how they actually use agents** without scheduling a 1-hour KT session (which the senior IC doesn't have time for, and the junior gets the polished demo version anyway).
- **Cannot defend agent-assisted code in review** with concrete "here's what the agent did and why" evidence. Falls back to "trust me" or rebuilds the context manually.

The substitute for all of this today is **nothing**. Or, more honestly: **vibes, scrollback, manual notes, and luck.** That's both the gap and the install-pitch.

## Target User & Narrowest Wedge

### Persona: Senior / Principal IC, AI-native

The "quark in the atom." The IC who:
- Makes shit happen on the team. Teams crumble without them.
- Codes daily AND mentors juniors AND surfaces roadblocks to leadership.
- Uses Claude Code / Cursor / pi-mono seriously (not as autocomplete — as workflow partner).
- Operates at both 50-person Series B startups and large enterprises (similar shape).
- Lives in a constant balancing act between IC delivery, mentorship overhead, and upward advocacy.

### Anti-users (explicit)

- **Delivery-metric pushers** who want pretty burndown charts regardless of quality. The "numbers look good" crowd.
- **Surveillance-style compliance/security teams** who want to monitor engineers, not give them sovereignty. Sovereignty ≠ surveillance, and a tool that does both does neither well.
- **Engineers who don't use agents seriously yet** (no pain to solve).
- **Engineers in locked-down enterprise environments** that can't run a daemon on the dev machine (the founders' own client, ironically — and that's fine).

### Wedge

**OpenStory v1 pitch (DM-able in one sentence):**

> "Every Claude Code and pi-mono session you've ever run, searchable from one place. Cross-machine, cross-project, never expires."

**Install hook:** the search demo. "What did I conclude last week? How did I solve that bug last month? What was that prompt that worked so well?" Multi-session-per-day means search has value on day one — not week three.

**First 60 seconds:** install, open the dashboard, OpenStory auto-discovers existing sessions, user immediately sees their own recent work rendered as a clean timeline and searchable from the search bar.

**What is NOT the install hook (but stays in the product):**
- The live cross-state co-observation Max uses — too novel to demo without context, requires both parties on OpenStory. Becomes the "look what's possible" gif in marketing, not the lead.
- DORA dashboards, evidence-for-PR-defense reports, learn-from-other-engineers gallery — all downstream features built on the event spine, not v1.

## Constraints

- **Two founders, no funding yet.** Need to self-fund or get to a small revenue base before quitting day jobs.
- **Open-source first.** MIT-style license for the personal-sovereignty version. Monetization happens later via team/org features (Obsidian / Linear / PostHog / Supabase pattern).
- **Cannot rely on the founders' enterprise client as customer #1.** Procurement will outlast runway.
- **Cannot do enterprise sales as a wedge** for the same reason.
- **Distribution channel is founder-led DM, not paid acquisition or HN virality.** The 20-50 senior-IC contacts are the entire seed list. There is no Plan B for distribution in v1.

## Premises

These are the load-bearing beliefs. If any of these turn out to be false, the strategy needs to change.

1. **Trust is the underlying value; observability produces it; sovereignty/portability is the retention story.** This is the OpenTelemetry analogy: nice-to-have until it isn't.
2. **Multiple sessions per day is the norm for the target user**, which means cross-session search has compounding value starting on day one (not week three).
3. **Senior/principal ICs are reachable via 1:1 trust networks**, not broadcast channels. Distribution is founder-led DMs to a known list of 20-50.
4. **The category is large enough to matter long-term.** Agent-native development is the new normal; the observability layer for it will become as standard as OpenTelemetry. No competitor has yet defined the category.
5. **Sovereignty drives acquisition for the post-platform-betrayal segment**, but for everyone else it drives retention, not first-install. The first-install pitch must be pain-named, not values-named.
6. **The free individual / paid team split is the right business model.** The team-features set must be designed to *not* attract the surveillance-buyer anti-user — even when they show up with money attached. This will be a real test of discipline.
7. **The 20-DM launch will produce a real signal.** Target: 5 install in 48 hours, 2 still using a week later. If it doesn't, the persona or the wedge is wrong.

## Approaches Considered

### Approach A: Continuity-as-wedge ("never lose a session again")

Lead with the loss-of-context pain. Install moment is "you just lost a session, now install OpenStory so it never happens again."

**Why rejected:** Reactive framing. Requires the user to have already felt the pain before they hear the pitch. Also depends on cross-machine sync being rock-solid out of the gate (which it is, but the pitch sells the disaster recovery rather than the daily value).

### Approach B: Observability-as-wedge ("see what your agent did")

Lead with the dashboard. Install moment is "see your last session in a clean timeline."

**Why rejected:** Table-stakes feature. Once they've seen one session, no return loop. Doesn't capture the cross-session value.

### Approach C: Search-as-wedge ("every session, searchable, forever") ✅ CHOSEN

Lead with the searchable history of every session, every prompt, every tool call. Install moment is "search across everything you've ever done with an agent."

**Why chosen:** Value compounds AND starts on day one (multiple sessions/day means the second-search payoff is hours away, not weeks). Search is unambiguous, demo-able in 60 seconds, and the technical implementation is already strong (FTS5 in `sqlite_store.rs`, both founders use it daily and call it "rockin'"). All other use cases (continuity, observability, learning, evidence) are emergent properties of having searchable history.

### Approach D: Live-watch-as-wedge ("pair-program on agent work in real-time")

Lead with the Max-watches-Katie-from-another-state behavior.

**Why rejected for v1 (kept in product):** Requires both parties on OpenStory — network effect blocks day-one value for solo installs. Visceral and shareable, but better as marketing artifact than install pitch.

### Approach E: Learning-gallery-as-wedge ("learn from senior engineers' real sessions")

Lead with a public gallery of curated sessions from named senior ICs.

**Why rejected for v1:** Two-sided market on day one. Founders would have to seed the gallery with their own sessions before it's interesting to anyone. Add later as a community feature.

## Recommended Approach

**Approach C: Search-first wedge.** Pitch is "every Claude Code / pi-mono session you've ever run, searchable from one place." Cross-machine, cross-project, persistent. The dashboard, the live-watch, the event spine, the sovereignty values — all live underneath the pitch. The pitch fits in a tweet, the demo fits in 60 seconds, the value lands on day one.

## Open Questions

1. **What does the team-features monetization look like in concrete terms?** Shared search across an engineering org? Cross-engineer learning? Evidence packets for managers? Each has a different anti-user risk — the manager-evidence one is closest to the surveillance line and needs careful design (or an explicit "this is for the IC, not the manager" framing).
2. **Is there a one-line install that includes the Claude Code hook setup?** Currently `just up-no-mongo` works, but a senior IC needs `brew install openstory && openstory init` (or similar) that handles binary install + watch dir + hook config in `~/.claude/settings.json` end-to-end. This is the only thing standing between today and DM-launchable.
3. **What's the OSS license?** MIT is the obvious default. Worth thinking about whether a copyleft variant (AGPL) protects the team-features moat better, or whether MIT + closed-source team features is cleaner.
4. **What's the support story when DM-launch users hit issues?** If 5 install and 2 of them have install bugs, do the founders fix them in real-time? (Recommended: yes, treat the first 20 as design partners, not customers — direct support is the feature.)
5. **At what point do you and Max go full-time?** Frame this as a milestone, not a date. Suggested: "10 paying teams, $X MRR, OR a clear lead investor who'll fund a year." Don't quit on hope.

## Success Criteria

**For the next 30 days:**

- v1 install one-liner ships (`brew` or `curl | sh` that handles binary + hook config).
- DM-launch to 20 senior-IC contacts (mix: consulting EAs, industry EAs, startup ICs).
- **Pass:** ≥5 install in 48h, ≥2 still using 7 days later.
- **Fail:** <5 install, OR none still using after a week → the wedge is wrong, the persona is wrong, or the install friction is too high. Return to /office-hours with the data.

**For the next 90 days (if 30-day passes):**

- 20 active weekly users across the 3 archetypes.
- At least 3 unprompted "I cannot work without this" testimonials.
- First conversation with a team about a paid tier (doesn't have to close — just has to happen).

## Distribution Plan

**v0 (next 2-3 weeks):** Polish install. Single self-contained install command. Brew tap or `curl | sh` that:
1. Installs the binary
2. Creates `~/.openstory` data dir
3. Writes the Claude Code hook config to `~/.claude/settings.json` (with confirmation)
4. Starts the daemon
5. Opens the dashboard

CI/CD: GitHub Actions builds binaries for macOS (Apple Silicon + Intel) and Linux on tag push. Homebrew tap auto-updates from release artifacts.

**v1 launch (next 30 days):** Founder-led DM blast to the 20-50 senior-IC contact list. Personal messages, not a mass send. Each one offers direct support during install. Track: who installed, who's still running it after 7 days, who shared it with a colleague.

**No paid acquisition. No HN launch yet** (save HN for when there are 50 active users and a "look at this real-time co-observation" demo to lead with). No conference talks yet. No Twitter launch. The seed list IS the launch.

**v2 (after 30-day proof):** Open the gates. Public website, HN launch, OSS marketing, gallery of consenting users' workflow patterns. By this point you have testimonials, retention numbers, and can name the category.

## Dependencies

- **Install one-liner:** estimated 2-3 days, blocking on nothing.
- **First-60-seconds dashboard polish:** can ship in parallel with install work, not blocking the launch.
- **Multi-machine docs:** "here's how to get all your machines feeding into one OpenStory instance" — needs a clean writeup. Currently works (Katie + Max have 4 machines live), but the setup story is undocumented.
- **Anti-user discipline in team-features design:** not a v1 dependency, but the moment you start designing team features (post-30-day), this needs to be the first design constraint, not the last.
